# 系统架构设计文档

## 1. 系统整体架构

### 核心定位

这不是一个单纯的"智能客服系统"，而是一个：**多渠道客资接入与微信承接调度系统**

### 四层架构

```
┌────────────────────────────────────────────────────────────┐
│  第一层：渠道接入层  (4个平台账号 × 2家店 = 4个机器人)    │
│  抖音-BOP / 抖音-龙膜 / 小红书-BOP / 小红书-龙膜          │
└────────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────────┐
│  第二层：智能接待层 (OpenClaw)                             │
│  - 真人化聊天 / 上下文理解                                 │
│  - 结构化提取 (车型、项目、预算)                           │
│  - 状态识别 (微信承接阶段)                                 │
│  - Call 业务中台 API                                       │
└────────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────────┐
│  第三层：业务中台 (小程序后台 + 数据库)                   │
│  - 线索表管理                                              │
│  - 门店硬隔离                                              │
│  - 销售轮转分配                                            │
│  - 1/3/10分钟SLA时效                                       │
│  - 报表统计                                                │
│  - 企业微信推送                                            │
└────────────────────────────────────────────────────────────┘
                          ↓
┌────────────────────────────────────────────────────────────┐
│  第四层：执行层                                             │
│  - 企业微信销售群 (BOP + 龙膜独立)                         │
│  - 销售微信客户端 (状态回写)                               │
│  - 小程序客户端 (店长查看)                                 │
└────────────────────────────────────────────────────────────┘
```

## 2. 模块职责划分

### 2.1 OpenClaw 智能接待层（负责什么）

**负责的工作：**
- ✅ 账号级机器人接待（独立上下文、独立话术）
- ✅ 像真人一样聊天（引导、反问、给建议）
- ✅ 引导留资（获取客户信息）
- ✅ 引导加微信（自然推进到微信沟通）
- ✅ 识别客户意图（车型、项目、预算、时间等）
- ✅ 输出结构化字段（通过 REST API）

**不负责的工作：**
- ❌ 主数据存储（交给小程序后台）
- ❌ 轮转底层逻辑（交给业务中台）
- ❌ 全局统计（交给业务中台）
- ❌ 时效追踪（交给业务中台）
- ❌ 企业微信推送（交给业务中台）

### 2.2 小程序后台（业务中台）

**核心职责：**
1. **线索存储** - 接收 OpenClaw 输出，存到数据库
2. **分配调度** - 根据轮转规则分配给销售
3. **时效管理** - 追踪 1/3/10 分钟规则
4. **统计报表** - 按门店、销售、渠道统计效率
5. **企业微信** - 推送通知、状态管理
6. **状态机** - 管理线索从接入到成交的全生命周期

### 2.3 企业微信执行层

**负责：**
- 通知销售新线索
- 推送首响/加微信提醒
- 时效超期升级提醒
- 日报推送
- 销售状态操作回写

## 3. 关键业务规则

### 3.1 账号与门店硬绑定

```
抖音-BOP 账号 →→→ 机器人 Bot-DY-BOP →→→ 仅分配给 BOP 店销售
                                       ↓
                              BOP店 3人轮转池
                              (Sales1, Sales2, Sales3)

抖音-龙膜 账号 →→→ 机器人 Bot-DY-LM →→→ 仅分配给 龙膜店销售
                                       ↓
                              龙膜店 2人轮转池
                              (Sales4, Sales5)
```

**核心原则：**
- 同一账号的所有线索，不会分配到其他门店
- 门店轮转池物理隔离，无交集

### 3.2 时效控制规则（1/3/10 分钟）

不是"成交升级规则"，而是"**加微信承接规则**"

```
时间轴：
├─ 0秒   ：线索到达
├─ 1分钟 ：首响 (首次回复)
│         销售 / 机器人必须有回复
│         触发企业微信提醒
│
├─ 3分钟 ：发起加微信
│         销售必须执行加微信动作
│         可能是：
│         • 让客户发微信号给我们
│         • 我们发微信号给客户
│         • 通过链接分享微信
│         触发企业微信动作提醒
│
└─ 10分钟 ：确认加微信结果
          销售必须更新加微信状态：
          • 已加上微信
          • 客户拒绝
          • 加微信失败
          • 继续平台沟通
          触发数据回写
```

### 3.3 加微信状态标准化

线索的"微信状态"字段有 7 个标准值：

```python
WECHAT_STATUS_ENUM = {
    'pending': '未发起加微信',
    'invited': '已发起加微信',
    'customer_sent': '客户已发微信号',
    'sales_sent': '销售已发微信号',
    'success': '已加上微信',
    'refused': '拒绝加微信',
    'failed': '加微信失败',
}
```

## 4. 数据流转

### 4.1 完整流程

```
【客户在抖音/小红书发消息】
         ↓
  OpenClaw 接收消息
  （为该账号配置的机器人）
         ↓
  机器人像真人一样回复
  （账号级人格，独立话术）
         ↓
  识别意图、提取字段
  (call OpenClaw API)
         ↓
  【调用后端 API】
  POST /api/v1/leads 
  {
    "platform": "douyin",
    "account_code": "DY-BOP-001",
    "customer_nickname": "...",
    "car_model": "宝马X5",
    "service_type": "隐形车衣",
    "budget_range": "8000-12000",
    "conversation_summary": "..."
  }
         ↓
  后端创建 lead_id
  根据 account_code 确定 store_code
  分配给该门店的销售轮转池
         ↓
  【推送企业微信】
  发送给 BOP 店群 + 对应销售
  "新线索：宝马X5隐形车衣，预算8k-1.2w"
  1分钟倒计时开始
         ↓
  【销售接手】
  1分钟内回复（可在微信或企业微信）
  更新首响时间
         ↓
  【3分钟内发起加微信】
  销售执行加微信动作
  （通过企业微信按钮或手工更新）
         ↓
  【10分钟内确认结果】
  销售更新 wechat_status
  流程结束，进入微信承接阶段
```

### 4.2 API 关键端点

```
POST   /api/v1/leads                  创建线索
GET    /api/v1/leads/{lead_id}        获取线索详情
PATCH  /api/v1/leads/{lead_id}        更新线索状态
GET    /api/v1/leads                  列表查询（支持筛选）

POST   /api/v1/leads/{lead_id}/assign 分配线索
POST   /api/v1/leads/{lead_id}/first-reply 记录首响
POST   /api/v1/leads/{lead_id}/wechat-invite 记录加微信
PATCH  /api/v1/leads/{lead_id}/wechat-status 更新微信状态

GET    /api/v1/stats/daily            日报统计
GET    /api/v1/stats/by-store         按门店统计
GET    /api/v1/stats/by-sales         按销售统计
GET    /api/v1/stats/sla              SLA统计
```

## 5. 第一阶段核心需求

### 5.1 必须完成的功能

**账号和机器人管理**
- [ ] 记录 4 个账号（抖音-BOP、抖音-龙膜、小红书-BOP、小红书-龙膜）
- [ ] 每个账号绑定机器人实例 ID
- [ ] 每个账号绑定门店 (store_code)

**线索接入与存储**
- [ ] OpenClaw 通过 API 创建线索
- [ ] 存储完整信息（客户昵称、车型、项目、预算、聊天摘要）
- [ ] 自动根据 account_code 设置 store_code

**分配逻辑**
- [ ] BOP 店：3人轮转（Sales1, Sales2, Sales3）
- [ ] 龙膜店：2人轮转（Sales4, Sales5）
- [ ] 定时轮转指针（每次分配后 +1）
- [ ] 线索分配后推送企业微信

**时效追踪**
- [ ] 记录 assigned_at（分配时间）
- [ ] 记录 first_reply_at（首响时间）
- [ ] 记录 wechat_invited_at（发起加微信时间）
- [ ] 记录 wechat_result_at（确认结果时间）
- [ ] 自动计算 SLA 状态（pass / fail）

**企业微信集成**
- [ ] 分配时推送到 BOP 店群 或 龙膜店群
- [ ] 发送格式：客户名+车型+项目+预算，附销售名
- [ ] 1分钟未回复时升级提醒
- [ ] 3分钟未发起加微信时升级提醒

**基础报表**
- [ ] 今日线索数（按门店、按渠道）
- [ ] 首响率（按门店）
- [ ] 加微信发起率（按门店）
- [ ] 加微信成功率（按门店）

### 5.2 暂不做（第二、三阶段）

- ❌ 小程序管理界面（后续开发）
- ❌ 销售绩效看板优化
- ❌ 渠道 ROI 分析
- ❌ 自动复盘和智能建议

## 6. OpenClaw 与后端的集成方案

### 6.1 接口协议

**OpenClaw → 后端：创建 / 更新线索**

```
POST /api/v1/leads
Content-Type: application/json

{
  "platform": "douyin",           # 接入平台：douyin / xiaohongshu
  "source_channel": "live",       # 渠道：live / invest / natural
  "account_code": "DY-BOP-001",   # 账号编码（关键！决定门店）
  "customer_nickname": "小王",
  "car_model": "宝马X5",
  "service_type": "隐形车衣",
  "budget_range": "8000-12000",
  "conversation_summary": "客户咨询BMW X5隐形车衣价格和施工周期",
  "customer_contact": "13800138000"  # 可选
}

返回：
{
  "lead_id": "lead_20260309_001",
  "store_code": "BOP",
  "assigned_to": "Sales1",
  "assigned_at": "2026-03-09T10:30:00Z",
  "status": "pending_first_reply"
}
```

### 6.2 状态同步

**后端 → OpenClaw：更新对话上下文**

```
POST /openclaw/api/sync-status
{
  "bot_instance_id": "Bot-DY-BOP",
  "customer_id": "...",
  "status": "assigned_to_sales",
  "sales_name": "李四",
  "sales_wechat": "lisi_wechat",
  "next_action": "Please contact sales via WeChat"
}
```

> 这样 OpenClaw 机器人可以感知销售已接手，自动更新对话策略

## 7. 部署架构

```
Frontend Layer:
├── 企业微信 (企业号接收企业微信通知)
├── 小程序 Admin (Vue3)
└── 销售微信客户端 (微信内操作)

Backend Layer:
├── FastAPI Server (Python)
├── PostgreSQL Database
└── OpenClaw Integration (REST API)

Message Queue (可选，第二阶段):
├── RabbitMQ or Redis (异步任务)
├── 定时轮转检查
┗─ 超期升级检查

External Services:
├── 企业微信 API
├── OpenClaw Bot API
└── WeChat Official Account API
```

## 8. 安全考虑

- 账号归属强校验（防止线索穿插）
- 门店隔离（不同店的销售无法看到其他店的线索）
- 时效数据不可修改（防止欺骗统计）
- 企业微信 token 加密存储
- API 请求签名验证
