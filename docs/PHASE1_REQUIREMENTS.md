# 第一阶段开发需求清单

## 项目概况

- **时间周期**：4-6周（根据团队规模）
- **核心目标**：构建多账号机器人 + 两店轮转 + SLA时效系统
- **完成标志**：线索从接入、分配、时效追踪到微信承接的完整闭环

## Phase 1 功能需求

### 1. 账号&机器人管理模块

**需求**：
- [ ] 支持创建/编辑/删除账号
- [ ] 每个账号绑定一个门店（BOP 或 LM）
- [ ] 每个账号绑定一个机器人实例 ID
- [ ] 账号状态管理（启用/禁用）

**涉及组件**：
- Backend: `POST /accounts`, `GET /accounts`, `PATCH /accounts/{code}`, `DELETE /accounts/{code}`
- Admin: 账号管理页面（表格+CRUD对话框）
- 数据库：account 表

**验收标准**：
- ✅ 可创建 4 个账号（抖音/小红书 × BOP/龙膜）
- ✅ 账号创建后自动绑定到门店
- ✅ 不允许跨店申请

### 2. 线索接入&存储模块

**需求**：
- [ ] OpenClaw 通过 API 创建线索
- [ ] 线索自动绑定到对应的账号、门店、机器人
- [ ] 存储结构化数据（车型、项目、预算、聊天摘要）
- [ ] 支持线索列表查询和详情查看

**涉及组件**：
- Backend: `POST /leads`, `GET /leads`, `GET /leads/{lead_id}`
- Admin: 线索中心（列表+详情+筛选）
- 数据库：lead 表

**验收标准**：
- ✅ OpenClaw 能成功创建线索
- ✅ 线索字段完整（客户昵称、车型、项目、预算、话术摘要）
- ✅ 支持按门店、渠道、日期筛选

### 3. 分配&轮转模块

**需求**：
- [ ] BOP 店：3 人独立轮转（Sales1, 2, 3）
- [ ] 龙膜店：2 人独立轮转（Sales4, 5）
- [ ] 线索自动分配给当前轮转销售
- [ ] 分配后轮转指针自动 +1
- [ ] 两店轮转池物理隔离（不会相互影响）

**涉及组件**：
- Backend: 分配引擎（在 `POST /leads` 创建后自动触发）
- Backend: `GET /allocation/status` 查询轮转状态
- Admin: 分配管理页面（查看轮转指针、人工干预等）
- 数据库：sales_allocation 表

**验收标准**：
- ✅ 每创建一个线索，自动分配给下一个销售
- ✅ BOP 店分配在 3 人循环内
- ✅ 龙膜店分配在 2 人循环内
- ✅ 可手工调整轮转指针（管理功能）

### 4. 时效追踪模块（1/3/10分钟）

**需求**：
- [ ] 记录线索创建时间（assigned_at）
- [ ] 记录首响时间（first_reply_at）
- [ ] 记录加微信发起时间（wechat_invited_at）
- [ ] 记录加微信结果时间（wechat_result_at）
- [ ] 自动计算 SLA 状态（pass / fail）
- [ ] 记录完整的时效变化历史（timeline）

**涉及组件**：
- Backend: 
  - `POST /leads/{lead_id}/first-reply` 记录首响
  - `POST /leads/{lead_id}/wechat-invite` 记录加微信
  - `PATCH /leads/{lead_id}/wechat-status` 更新微信状态
- Backend: 定时任务检查 SLA 超期
- Admin: 线索详情页展示时效变化
- 数据库：lead_timeline 表

**验收标准**：
- ✅ 线索时间戳自动记录
- ✅ SLA 状态正确计算（1、3、10分钟）
- ✅ 时间戳不可修改（数据完整性）
- ✅ Timeline 完整记录每次变化

### 5. 企业微信推送模块

**需求**：
- [ ] 线索分配时，推送到销售个人 + 店群
- [ ] 推送格式：客户昵称 + 车型 + 项目 + 预算 + 销售名
- [ ] 1 分钟未首响时：升级提醒
- [ ] 3 分钟未加微信时：升级提醒
- [ ] 10 分钟未确认结果时：最终提醒
- [ ] 支持销售在企业微信中点按钮回写状态

**涉及组件**：
- Backend: 
  - `POST /wecom/notify-new-lead` 新线索通知
  - `POST /wecom/notify-sla-timeout` SLA 超期提醒
  - `POST /wecom/update-button` 接收企业微信按钮回调
- Backend: 定时任务检查 SLA，触发推送
- 数据库：wecom_log 表（记录推送日志）

**验收标准**：
- ✅ 线索分配时立即推送到销售 + 店群
- ✅ 超期提醒准时推送
- ✅ 企业微信按钮回调正确更新状态

### 6. 基础报表&统计模块

**需求**：
- [ ] 日报统计（按日期、门店、渠道）
- [ ] 线索数统计
- [ ] 首响率（首响线索数 / 总线索数）
- [ ] 加微信发起率（发起加微信线索数 / 总线索数）
- [ ] 加微信成功率（成功线索数 / 总线索数）
- [ ] SLA 通过率（按 1/3/10 分钟分别统计）
- [ ] 按销售统计（每个销售的分配数、首响率、成功率）

**涉及组件**：
- Backend: 
  - `GET /stats/daily` 日报
  - `GET /stats/by-sales` 按销售统计
  - `GET /stats/sla` SLA 统计
- Backend: 定时任务生成 daily_stats 聚合数据
- Admin: 统计报表页面（图表 + 表格）
- 数据库：daily_stats 表

**验收标准**：
- ✅ 日报数据准确，每日 00:00 自动聚合
- ✅ 各项数据可在 Admin 中查看
- ✅ 支持按日期范围查询
- ✅ 可导出 Excel 报表

## 技术栈选型

### 后端（Backend）

- **框架**：Python FastAPI
- **ORM**：SQLAlchemy 2.0
- **数据库**：PostgreSQL 14+
- **认证**：JWT (PyJWT)
- **企业微信 SDK**：企业微信官方 SDK
- **任务调度**：APScheduler
- **测试**：pytest
- **API 文档**：Swagger (FastAPI 内置)

### 前端（Admin）

- **框架**：Vue 3 + Vite
- **UI 组件库**：Element Plus
- **状态管理**：Pinia
- **HTTP 客户端**：axios
- **图表库**：ECharts
- **表格**：Element Plus Table

### 小程序（MiniProgram）

- **框架**：原生微信小程序
- **UI 框架**：可选 (原生或 WXS)
- **HTTP**：wx.request

## 部署架构

### 开发环境

```
macOS/Linux 本地：
├── 后端：python app/main.py (http://localhost:8000)
├── Admin：npm run dev (http://localhost:5173)
├── PostgreSQL：docker-compose up -d
└── 小程序：微信开发者工具
```

### 生产环境（第二阶段）

```
Docker + K8s:
├── Backend 服务
├── PostgreSQL 数据库
├── Nginx 反向代理
├── 企业微信 webhook
└── 小程序云托管（可选）
```

## 验收标准

### 功能验收

- ✅ 4 个账号可正常创建和管理
- ✅ 线索从 OpenClaw 成功进入系统
- ✅ 线索自动分配到两店销售（轮转正确）
- ✅ 1/3/10 分钟时效规则自动追踪
- ✅ 企业微信推送准时且格式正确
- ✅ 日报统计数据准确
- ✅ 系统稳定性（无崩溃、无数据丢失）

### 性能验收

- ✅ 线索创建响应时间 < 200ms
- ✅ 企业微信推送延迟 < 1s
- ✅ 日报生成时间 < 5s
- ✅ 能支持日均 500+ 线索量

### 安全验收

- ✅ API 认证正确
- ✅ 权限隔离（销售只能看自己的线索）
- ✅ 时效数据不可修改
- ✅ 企业微信 token 加密存储

## 关键风险和应对

| 风险 | 影响 | 应对方案 |
|------|------|--------|
| OpenClaw API 文档不完整 | 集成延期 | 提前沟通 OpenClaw 方，准备 Mock 数据 |
| 企业微信权限申请慢 | 推送功能延期 | 预留接口，先用日志系统替代 |
| 数据量突增 | 性能问题 | 提前规划索引，实施分库分表（后期） |
| 微信小程序审核 | 上线延期 | 小程序与后端分离，后端先上线 |
