# 架构设计文档

## 1. 系统架构概览

### 1.1 分层架构

```
┌─────────────────────────────────────────────────┐
│         前端展示层 (Admin + 小程序)              │
├─────────────────────────────────────────────────┤
│    业务API层 (FastAPI)                          │
│  - 账号管理 | 线索管理 | 分配轮转 | 报表统计   │
├─────────────────────────────────────────────────┤
│    业务服务层                                   │
│  - 账号服务 | 线索服务 | 分配引擎 | SLA引擎   │
├─────────────────────────────────────────────────┤
│    数据访问层 (SQLAlchemy ORM)                  │
├─────────────────────────────────────────────────┤
│    数据库层 (PostgreSQL)                        │
└─────────────────────────────────────────────────┘
        ↕
    ┌─────────┬─────────┬─────────┐
    │OpenClaw │企业微信  │抖音/小红书
    │智能接待  │通知推送  │平台账号
    └─────────┴─────────┴─────────┘
```

### 1.2 组件职责

| 层级 | 组件 | 职责 |
|-----|------|------|
| OpenClaw层 | Bot-DY-BOP等 | 真人化聊天、意图识别、提取、推进 |
| 业务中台 | FastAPI后端 | 分配、轮转、时效、统计 |
| 执行层 | 企业微信 | 通知销售、群消息、状态回写 |
| 展示层 | Admin+小程序 | 管理界面、线索展示、操作界面 |

## 2. 数据流

### 2.1 客户来源→微信承接完整流

```
【客户来源】
抖音直播/投流/自然流 → OpenClaw Bot → 平台私信
小红书 → OpenClaw Bot → 平台私信

【中台处理】
1. 平台消息进入 → 线索入库
2. 智能提取 → 结构化字段
3. 账号硬绑定 → 门店路由
4. 轮转池分配 → 触发销售通知
5. SLA计时 → 1分钟首响
             → 3分钟加微信动作
             → 10分钟确认结果

【企业微信】
→ 通知销售个人
→ 同时通知店群
→ 超时升级
→ 完成状态回写

【微信承接】
→ 销售发微信二维码
→ 客户扫码加好友
→ 微信沟通→成交
```

### 2.2 时序图：1/3/10分钟规则

```
时间线:
T+0s  ← [线索入库] lead_created
T+60s ← [1m超时检查] 首响 (销售未回复→升级提醒)
T+3m  ← [3m超时检查] 加微信动作 (未发起→升级提醒)
T+10m ← [10m超时检查] 确认结果 (未确认→再次提醒)
T+后续 ← 微信沟通→成交
```

## 3. 多账号机器人隔离设计

### 3.1 账号→机器人映射

```
┌──────────────┬───────────┬─────────────┐
│ 平台         │ 账号      │ 机器人实例  │
├──────────────┼───────────┼─────────────┤
│ 抖音         │ BOP账号   │ Bot-DY-BOP  │
│ 抖音         │ 龙膜账号  │ Bot-DY-LM   │
│ 小红书       │ BOP账号   │ Bot-XHS-BOP │
│ 小红书       │ 龙膜账号  │ Bot-XHS-LM  │
└──────────────┴───────────┴─────────────┘
```

### 3.2 隔离规则

1. **界面隔离**：每个Bot有自己的人设、话术、推进节奏
2. **数据隔离**：线索天然属于某个账号，不会混淆
3. **状态隔离**：对话状态独立维护，无交叉污染
4. **输出隔离**：后续分配、企业微信通知都按账号源头追溯

## 4. 两店轮转分配设计

### 4.1 轮转机制

```
【BOP店】
销售轮转池: [销售A] → [销售B] → [销售C] → [销售A]...
当前指针：销售A
线索入库 → 指针指向销售A → 分配 → 指针移到销售B

【龙膜店】
销售轮转池: [销售D] → [销售E] → [销售D]...
当前指针：销售D
```

### 4.2 核心字段

```
account_code: "DY_BOP"
store_code: "BOP"
assigned_sales_id: "sales_001"
assigned_group: "BOP_GROUP"
assigned_at: "2024-01-15 10:30:45"
```

## 5. SLA时效管理

### 5.1 状态机

```
[线索创建] 
    ↓ (开始1分钟计时)
[待首响] 
    ↓ (销售回复后)
[已首响]
    ↓ (开始3分钟计时)
[待加微信动作]
    ↓ (销售发起加微信后)
[已发起加微信]
    ↓ (开始10分钟计时)
[待确认加微信]
    ↓ (销售确认加上或失败)
[已加微信] / [失败] / [拒绝]
```

### 5.2 字段定义

```
sla_1m_status:  # "pending" / "success" / "timeout"
sla_3m_status:  # "pending" / "success" / "timeout"
sla_10m_status: # "pending" / "success" / "timeout"

first_reply_at:      # 首响时间
wechat_invited_at:   # 发起加微信时间
wechat_result_at:    # 确认加微信时间
```

## 6. OpenClaw集成

### 6.1 消息流

```
[平台消息]
    ↓
[OpenClaw API] ← 发送消息 + 历史context
    ↓
[OpenClaw返回] 
    {
      "response": "真人化回复内容",
      "intent": "车衣咨询",
      "car_model": "宝马X5",
      "service_type": "车衣+隐形车衣",
      "budget_range": "5000-10000",
      "wechat_status": "未发起加微信",
      "next_action": "发起加微信"
    }
    ↓
[后台入库]
    - 保存在线索的 structured_data 字段
    - 基于数据触发分配和SLA
```

### 6.2 预留接口

- `POST /api/openclaw/chat` - 发送聊天消息给OpenClaw
- `POST /api/openclaw/webhook` - 接收OpenClaw回调
- `GET /api/openclaw/mock/leads` - 获取Mock数据

## 7. 企业微信集成

### 7.1 推送类型

1. **新线索通知** → 个人 + 店群
2. **超时升级** → 店长 + 管理群
3. **状态回写** → 销售确认后回写

```json
{
  "recipient_type": "person|group",
  "recipient_id": "sales_001",
  "message_type": "new_lead|sla_timeout|status_update",

    "car_model": "宝马X5",
    "source": "抖音",
    "action": "请在1分钟内首响"
  }
}
```

## 9. 总结

- **OpenClaw** = 智能接待层
- **FastAPI后端** = 业务中枢（分配、时效、统计）
- **小程序+Admin** = 操作和展示
- **企业微信** = 执行和通知
- **PostgreSQL** = 唯一数据源

这样的架构下，后续可以灵活扩展：换机器人、加账号、加门店都不会影响核心逻辑。
