# 数据库优先：详细任务清单（可执行版）

> 目标：先解决“数据库缺失”这个核心问题，再推进安全、财务、运营功能。

## 0. 优先级重排（最终执行顺序）
1. **P0-1 数据库与单一数据真源（最高优先级）**
2. P0-2 安全基线（密码哈希、token 生命周期、内部接口鉴权强制）
3. P0-3 对象存储（图片持久化 URL）
4. P1 财务真实对接与回写
5. P2 报表、告警与运营体验

---

## 1. P0-1 数据库与单一数据真源（最高优先级）

## 1.1 里程碑与时间（建议 3 周）
- **W1：数据库落地 + 基础表结构 + 只读接入**
- **W2：双写改造 + 迁移校验 + 增量 API**
- **W3：切主与回滚演练 + 关停 JSON 主写**

## 1.2 详细任务拆分（WBS）

### A. 方案设计（Day 1-2）
- [ ] A1. 选型确认：PostgreSQL 15（主）/ MySQL 8（备选）
- [ ] A2. 部署方式确认：同机部署 or 托管云数据库
- [ ] A3. 命名规范确认：表名、索引名、审计字段规范
- [ ] A4. 数据保留策略：订单永久、日志 180 天归档
- [ ] A5. 产出 ER 图与字段字典（评审通过）

### B. 数据库建模（Day 3-5）
- [ ] B1. 建 `users`（账号、角色、密码哈希、状态、最后登录）
- [ ] B2. 建 `orders`（订单主表、状态、金额、业务类型、版本号）
- [ ] B3. 建 `order_dispatches`（派工记录，支持多次改派）
- [ ] B4. 建 `order_work_parts`（施工部位与提成明细）
- [ ] B5. 建 `followups`（7/30/60/180 节点、状态）
- [ ] B6. 建 `finance_sync_logs`（请求、响应、结果、重试次数）
- [ ] B7. 建 `audit_logs`（谁在何时改了什么）
- [ ] B8. 补核心索引（订单号、更新时间、状态、销售、预约时间）
- [ ] B9. 建唯一约束（`orders.order_id`）
- [ ] B10. 建乐观锁字段（`orders.version`）

### C. 后端数据访问层改造（Week 1）
- [ ] C1. 抽象 repository 层，隔离 JSON 读写
- [ ] C2. 新增 DB repository（users/orders/dispatch/followup/finance）
- [ ] C3. API 切到“读 DB 优先，JSON 兜底”模式（灰度）
- [ ] C4. 新增健康检查 `/health/db`（连通与延迟）
- [ ] C5. 增加 SQL 超时和连接池参数

### D. 数据迁移（Week 2）
- [ ] D1. 编写迁移脚本：`users.json -> users`
- [ ] D2. 编写迁移脚本：`orders.json -> orders + 子表`
- [ ] D3. 编写迁移脚本：`finance-sync-log.json -> finance_sync_logs`
- [ ] D4. 数据清洗规则（空值、状态别名、非法手机号）
- [ ] D5. 迁移后校验脚本（数量校验、金额汇总校验、抽样 diff）
- [ ] D6. 首次全量迁移演练（测试环境）
- [ ] D7. 二次增量迁移演练（测试环境）

### E. 同步机制重构（Week 2）
- [ ] E1. 新增增量拉取 API：`GET /api/v1/orders?updatedAfter=`
- [ ] E2. 新增 patch API：`PATCH /api/v1/orders/{id}`
- [ ] E3. 后端冲突检测（version 不一致返回 409）
- [ ] E4. 小程序改造：全量覆盖 -> 增量拉取 + patch 提交
- [ ] E5. 管理台改造：编辑提交带 version
- [ ] E6. 冲突提示 UI（“数据已被他人更新，请刷新后重试”）

### F. 双写与切换（Week 3）
- [ ] F1. 上线双写：写 DB + 写 JSON（只保留一周）
- [ ] F2. 监控双写一致性（每小时对比任务）
- [ ] F3. 只读切换：查询全部走 DB
- [ ] F4. 主写切换：写入全部走 DB，JSON 停写
- [ ] F5. 保留 JSON 快照归档（只读备查）
- [ ] F6. 输出切换报告（时间点、影响范围、回滚点）

### G. 验收与回滚（Week 3）
- [ ] G1. 压测：并发编辑 50 用户，无覆盖写
- [ ] G2. 核验：订单总数、状态分布、金额分布与切换前一致
- [ ] G3. 演练：DB 故障回滚流程（15 分钟内）
- [ ] G4. 发布 Go/No-Go 检查清单签字

## 1.3 验收标准（必须全部满足）
- [ ] 数据一致：迁移后订单数量误差 = 0
- [ ] 并发安全：冲突写全部返回 409，不发生静默覆盖
- [ ] 性能：订单列表 P95 < 300ms（同内网条件）
- [ ] 稳定：上线后 7 天无 P1 数据事故

## 2. P0-2 安全基线（数据库切主后立刻做）

### 任务清单
- [ ] S1. 密码改为 bcrypt/argon2 哈希（含迁移策略）

- [ ] S5. 登录失败限流与锁定
- [ ] S6. 审计“登录/改密/重置密码”关键动作

### 验收
- [ ] 明文密码为 0
- [ ] 未鉴权请求全部 401/403
