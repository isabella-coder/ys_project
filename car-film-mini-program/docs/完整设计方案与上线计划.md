# 汽车门店系统完整设计方案与上线计划（v2）


## 0. 优先级更新（根据最新反馈）

> 最新共识：**数据库建设是最高优先级**，先做数据库与单一真源，再做其他功能。

- TOP1：数据库与单一真源（替换 JSON 主存储）
- TOP2：安全基线
- TOP3：图片对象存储
- TOP4：真实财务对接与回写
- TOP5：报表与运营提效

详细可执行任务见：`docs/数据库优先详细任务清单.md`。

---

## 1. 目标与结论

### 1.1 业务目标
- 支撑 **小程序前台（贴膜+洗车）** 与 **电脑端管理后台** 的统一订单流。
- 实现从下单、派工、施工、交付、回访到财务入账的端到端闭环。
- 具备可上线能力：稳定域名、可追溯、可回滚、可扩容、可审计。

### 1.2 现状结论（基于代码与数据）
- 架构确实是 **两个前端 + 一个后端服务**：
  - 小程序前端：业务入口与订单操作。
  - 电脑端前端：管理台（登录/订单/派工/回访/财务日志）。
  - Python 后端：提供 API + 托管管理台静态资源。
- 当前可视为“可内测版”，核心流程已可跑通，但离“生产稳定版”仍有关键缺口。

## 2. 当前系统架构（As-Is）

## 2.1 终端与访问
- 小程序：负责前台业务操作、图片上传、触发财务同步。
- 电脑端管理台：浏览器访问后台系统，进行运营与管理动作。

## 2.2 服务端
- 单体 Python 服务：
  - 对外提供订单、派工、回访、财务日志、用户权限 API。
  - 提供后台登录与角色鉴权。
  - 使用 JSON 文件作为持久化存储。

## 2.3 数据存储
- 小程序本地存储：订单、产品目录、当前账号、运行时财务地址。
- 服务端本地 JSON：`orders.json`、`users.json`、`finance-sync-log.json`。

## 2.4 关键流程（现有）
1. 小程序下单 -> 写本地订单 -> 与服务端做全量合并同步。
2. 派工/施工/完工 -> 更新订单 -> 触发财务同步。
3. 财务同步接口目前是桥接式日志入库（非真实财务系统深度集成）。

## 3. 缺失项盘点（Gap Analysis）

## 3.1 P0（上线前必须补齐）
1. **统一数据真源缺失**
   - 现为“小程序本地 + 服务端合并”的模式，存在多人并发覆盖风险。
   - 目标：服务端单一真源，客户端只做缓存与展示。

2. **公网与网络稳定性不足**
   - trial/release 仍依赖临时域名形态，生产风险高。
   - 目标：正式 HTTPS 域名 + 证书 + 反向代理 + 健康检查。

3. **认证与安全基线不足**
   - 管理端 token 在内存，重启失效且不可审计。
   - 密码为明文校验，不满足生产规范。
   - 内部接口 token 机制存在“未配置可放行”模式，建议收敛。

4. **媒体资产管理不足**
   - 图片大量是临时路径，跨设备与长期留存风险高。
   - 目标：对象存储（OSS/COS/S3）+ 持久 URL + 访问策略。

## 3.2 P1（运营效率）
1. 后台缺少洗车专属经营视图（产能、时段占用、超时单）。
2. 缺少审计日志（谁在何时修改了什么）。
3. 财务对接仍偏“日志同步”，缺少字段映射、回写与对账闭环。

## 3.3 P2（治理与增长）
1. 历史数据清洗与主数据规范（人员名、门店名、车型格式）。
2. 经营报表导出（日/周/月销售、提成、工位利用率）。
3. 自动化告警（同步失败、超容量、回访逾期）。

## 4. 目标架构（To-Be）

## 4.1 总体原则
- **后端单一真源（SSOT）**：订单状态仅以后端数据库为准。
- **前端无状态化**：小程序和管理台仅保留必要缓存，不做主数据覆盖。
- **可观测与可追溯**：每一次状态变更均有审计记录与可回放事件。
- **安全默认开启**：最小权限、默认拒绝、密钥轮换。

## 4.2 推荐技术蓝图（保持演进平滑）
1. **应用层**
   - 保持现有 Python 服务，先做“结构化重构”（模块拆分）再考虑框架升级。
   - 接口分层：
     - `public api`（后台管理）
     - `mini api`（小程序业务）
     - `internal api`（财务与内部任务）

2. **数据层**
   - 从 JSON 升级到关系型数据库（建议 PostgreSQL / MySQL）。
   - 核心表：
     - `users`
     - `orders`
     - `order_dispatches`
     - `order_commissions`
     - `followups`
     - `finance_sync_logs`
     - `audit_logs`

3. **对象存储层**
   - 上传凭证、施工图、尾款图统一走对象存储。
   - 订单仅保存 URL 与元数据（哈希、上传时间、上传人）。

4. **任务与集成层**
   - 财务同步改为“异步任务队列 + 重试机制 + 死信处理”。
   - 增加对账任务（按日比对订单状态与财务状态）。

5. **安全层**
   - 密码哈希（bcrypt/argon2）。
   - JWT/会话令牌持久化（含过期与刷新）。
   - CORS 白名单与内部接口强制 token。
   - 接口限流、登录失败锁定、关键操作二次确认（可选）。

6. **运维层**
   - Nginx + HTTPS + systemd/容器化部署。
   - 结构化日志 + 指标监控 + 错误告警（飞书/企业微信）。

## 5. 关键模块设计

## 5.1 订单域模型（建议）
- 状态机：`CREATED -> DISPATCHED -> IN_PROGRESS -> DELIVERY_APPROVED -> COMPLETED / CANCELLED`
- 每次状态变更必须记录：操作人、来源端、时间、前后值。
- 订单版本号（`version`）用于并发控制，避免覆盖写。

## 5.2 同步策略改造
- 当前：拉远端 + 本地合并 + 全量 push。
- 目标：
  - 客户端按 `updatedAfter` 拉增量。
  - 客户端提交 patch（仅变更字段）。
  - 服务端按版本号做冲突检测，返回冲突详情。

## 5.3 财务集成策略
- 建立标准财务事件：
  - `ORDER_CREATED`
  - `ORDER_DISPATCH_UPDATED`
  - `ORDER_DELIVERY_CONFIRMED`
  - `ORDER_COMPLETED`
  - `ORDER_CANCELLED`
- 映射配置化：字段映射表 + 枚举映射表。
- 回写机制：记录财务流水号、入账状态、失败原因、重试次数。

## 5.4 权限模型
- 角色：`manager/sales/technician/finance`
- 细粒度权限：资源（订单、派工、回访、财务）+ 动作（读/写/审批/导出）
- 支持按门店或组织隔离（为后续多门店扩展预留）。

## 6. 分阶段实施计划（建议 8~10 周）

## 阶段 A（第 1~2 周）：上线基线
- 任务：
  1. 正式域名、HTTPS、Nginx、进程守护。
  2. 密码哈希改造、token 过期机制。
  3. 内部接口强制 token（无 token 一律拒绝）。
  4. 备份策略（每日快照 + 保留 7/30 天）。
- 交付标准：
  - 具备稳定外网访问与最小安全合规。

## 阶段 B（第 3~5 周）：数据真源改造
- 任务：
  1. 引入数据库并完成 JSON -> DB 迁移脚本。
  2. 新增增量同步 API 与版本控制。
  3. 小程序改为“服务端为准”的读写模式。
- 交付标准：
  - 多人并发编辑不覆盖，订单一致性可验证。

## 阶段 C（第 6~7 周）：财务闭环与审计
- 任务：
  1. 财务真实接口字段映射。
  2. 异步队列 + 重试 + 对账。
  3. 审计日志（操作轨迹）上线。
- 交付标准：
  - 财务状态可回写，失败可追踪并可重试。

## 阶段 D（第 8~10 周）：运营提效
- 任务：
  1. 洗车专属看板 + 销售/产能报表导出。
  2. 自动告警规则上线。
  3. 历史数据清洗与主数据治理。
- 交付标准：
  - 日常运营可视化与异常响应机制可用。

---

## 7. 验收指标（KPI）
- 稳定性：API 可用性 >= 99.9%。
- 一致性：订单写冲突自动识别，覆盖写事故为 0。
- 财务成功率：自动同步成功率 >= 98%，重试后 >= 99.5%。
- 安全：密码明文存储为 0，内部接口未鉴权访问为 0。
- 运营效率：派工冲突发现提前率、回访逾期率下降可量化。

---

## 8. 风险与应对
1. **迁移风险（JSON -> DB）**
   - 预案：灰度双写 + 回滚脚本 + 全量校验。
2. **真实财务接口不稳定**
   - 预案：队列削峰、指数退避、死信告警。
3. **终端版本不一致**
   - 预案：API 版本化 + 最低客户端版本策略。
4. **业务规则持续变化**
   - 预案：规则配置化（提成、回访节点、容量阈值）。

1. 先完成 P0 中“安全基线 + 正式域名 + 数据真源方案评审”。
2. 与财务系统确认字段字典、幂等规则、回写协议。
3. 用 1 周做“数据库迁移 PoC + 增量同步 API PoC”，验证后再全量切换。

> 一句话总结：你们已经具备“可内测的业务闭环”，下一阶段重点不是继续堆页面，而是尽快完成 **生产化三件套：数据真源、安全基线、真实财务闭环**。
