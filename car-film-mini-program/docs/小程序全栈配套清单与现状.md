## 2. 当前项目模块映射

### 2.1 小程序前端
- 入口与页面注册：`app.json`
- 页面目录：`pages/`
- 订单本地存储与同步：`utils/order.js`
- 财务同步：`utils/finance-sync.js`
- 同步配置（Base URL / API Token）：`config/finance.config.js`

### 2.2 管理后台（Web）
- 后台页面：`admin-console/web/index.html`
- 前端逻辑：`admin-console/web/app.js`
- 样式：`admin-console/web/styles.css`

### 2.3 后端服务
- 主服务：`admin-console/server.py`
- 核心接口：
  - `GET /health`
  - `GET /api/v1/store/internal/orders`
  - `POST /api/v1/store/internal/orders/sync`
  - `POST /api/v1/store/internal/work-orders/sync`
  - `POST /api/v1/store/login` / `POST /api/v1/store/logout`

### 2.4 数据与迁移
- 数据文件：`admin-console/data/*.json`
- SQL 迁移交付：`admin-console/sql/migrations/delivery/`

### 2.5 规范与流程
- Spec 仓库：`specs/00-governance` ~ `specs/05-release`
- 系统设计基线：`SYSTEM_DESIGN_TASK_BREAKDOWN_SPEC.md`

## 3. 现阶段还缺什么（按优先级）

### P0（立即补齐）
1. 线上同步鉴权闭环稳定化（确保体验版/正式版都配置正确 Base URL + API Token）。
2. 发布前固定执行冒烟测试并留结果（`scripts/smoke_api.sh`）。
3. 发布入口收敛为一套标准流程，避免多人使用不同脚本。

### P1（1-2 周补齐）
1. 建立接口回归测试与失败告警（至少登录、拉单、推单、冲突、财务同步）。
2. 增加监控/告警（接口失败率、5xx、数据库连接异常）。
3. 完成审计日志写入与查询链路（当前表结构已具备，服务写入仍需完善）。
4. 对象存储附件链路落地（当前仅有 attachments 表结构）。

### P2（后续优化）
1. CI/CD 自动化（Push 后自动测试、自动构建、自动部署）。
2. 离线队列和弱网重放机制升级（目前为基础同步+冲突处理）。
3. 发布证据自动归档（reports 与 go/no-go 记录统一管理）。

## 4. 推荐执行顺序（可直接照做）

1. **先稳线上联通**
   - 在小程序首页“订单同步配置”填写线上 Base URL 与 API Token。
   - 点击“立即同步订单”，确认状态为“同步正常”。
2. **建立发版前固定动作**
   - 启动后端后执行：
     - `export INTERNAL_API_TOKEN="<token>"`
     - `BASE_URL="http://127.0.0.1:8000" bash scripts/smoke_api.sh`
3. **冻结发布入口**
   - 在团队内明确一套脚本与文档作为唯一发布标准。
4. **补齐质量与运维能力**
   - 按 `specs/04-test` 与 `specs/05-release` 把 gate 项逐条落地并打勾。

## 5. 完成标准（DoD）

满足以下条件可视为“可持续运营”：
1. 体验版与正式版同步均稳定（非本地环境下可验证）。
2. 每次发布前有固定冒烟结果与记录。
3. 发生故障时有明确告警、定位路径、回滚路径。
4. Spec 状态从 `IN_REVIEW` 推进到 `APPROVED/IMPLEMENTED/VERIFIED`。
